# LUKHAS 3-Layer Tone System - Quick Start Guide

**For New Agents & Contributors**: Start here to understand LUKHAS's distinctive content approach

---

## The Core Concept (60-Second Version)

**The LUKHAS 3-Layer Tone System is a SEQUENTIAL NARRATIVE FLOW, not a tone menu.**

```
Layer 1 (Poetic Hook) → Layer 2 (Academic Depth) → Layer 3 (User-Friendly Bridge)
```

**Purpose**: Create a complete narrative journey from emotional engagement through intellectual understanding to practical application.

**When to Use**: **REQUIRED** for all public-facing documentation, websites, marketing, and concept introductions.

**When to Exclude**: Internal dev docs, API references, official specifications where precision > poetry.

---

## The Three Layers Explained

### Layer 1: Poetic/Keatsian Hook (2-4 paragraphs)

**Goal**: Capture attention, ignite curiosity, create emotional resonance

**Style**:
- Eye-catching, metaphorical, beautiful
- Keatsian romanticism (truth through beauty)
- Rich imagery and philosophical depth
- Make readers **feel** before they **think**

**Example Opening**:
> *"Imagine consciousness as an infinite tapestry, woven from threads of experience that fold back upon themselves in patterns too intricate for linear comprehension..."*

**NOT**:
> ❌ "Our innovative memory solution leverages advanced algorithms..."

---

### Layer 2: Academic Depth (6+ paragraphs)

**Goal**: Provide rigorous technical grounding and intellectual depth

**Style**:
- Peer-review ready, academically rigorous
- Complete technical accuracy
- Theoretical frameworks, quantitative metrics
- Formal structure: definitions → mechanisms → implications → evidence

**Example**:
> *"The fold-based memory architecture implements a hierarchical data structure inspired by topological folding in high-dimensional spaces. Each 'fold' is formally defined as a persistent data structure (in the Okasaki sense)..."*

**NOT**:
> ❌ "It uses advanced algorithms to make memory fast..."

---

### Layer 3: User-Friendly Bridge (Multiple examples)

**Goal**: Make complex concepts accessible and actionable

**Style**:
- Accessible language, practical examples
- "What does this mean for me? How do I use this?"
- Real code, real scenarios, real benefits
- Concrete over abstract

**Example**:
> *"So what does all this mean for you as a developer? Here's a simple example: Imagine you're building a conversation interface..."*

**NOT**:
> ❌ Just repeating Layer 2 in simpler words (must add practical value)

---

## Decision Tree: Should I Use the 3-Layer System?

```
Is this content PUBLIC-FACING?
├─ Yes → Is it introducing a CONCEPT?
│  ├─ Yes → ✅ USE 3-LAYER SYSTEM (required)
│  └─ No → Is it marketing/educational?
│     ├─ Yes → ✅ USE 3-LAYER SYSTEM
│     └─ No → ⚠️ Use judgment
└─ No → Is this internal/dev documentation?
   ├─ Yes → ❌ DO NOT USE (precision required)
   └─ No → ⚠️ Use judgment
```

---

## Usage Guidelines (Crystal Clear)

### ✅ ALWAYS USE 3-Layer System:

- Website landing pages
- Blog posts introducing concepts
- Product documentation (concept sections)
- Marketing materials
- Educational tutorials
- Community announcements (new features)
- White papers for mixed audiences
- Thought leadership content

### ❌ NEVER USE 3-Layer System:

- API reference documentation
- Code comments & docstrings
- Technical specifications
- Implementation guides for developers
- Debugging guides
- Internal wikis
- Official compliance documents
- Security advisories

### ⚠️ JUDGMENT CALL:

- **Technical docs with concept intros**: Use 3-layer for intro section, then pure Layer 2 for details
- **Academic papers**: Layer 2 only (unless for broader audiences)
- **Hybrid contexts**: When introducing concepts even in technical contexts, consider 3-layer intro

---

## High Verbosity Preference

**CRITICAL**: Do NOT rush the layers. Each needs space to accomplish its purpose.

- **Layer 1**: 2-4 FULL paragraphs (let poetry breathe)
- **Layer 2**: 6+ paragraphs (complete technical depth)
- **Layer 3**: 2-3 different examples from different angles

**Better**: Deep 3-layer treatment of ONE concept
**Worse**: Rushed 1-paragraph layers across many concepts

## Quick Quality Checklist

Before publishing public-facing content, verify:

- [ ] **Layer 1 Present**: 2-4 paragraphs of poetic introduction
- [ ] **Layer 1 Quality**: Uses metaphors, creates imagery, feels beautiful
- [ ] **Transition 1→2**: Smooth bridge from poetry to precision
- [ ] **Layer 2 Present**: 6+ paragraphs of technical depth
- [ ] **Layer 2 Quality**: Peer-review ready, complete technical accuracy
- [ ] **Transition 2→3**: Smooth bridge from theory to practice
- [ ] **Layer 3 Present**: Multiple practical examples
- [ ] **Layer 3 Quality**: Real code, real scenarios, immediately useful
- [ ] **High Verbosity**: Each layer has adequate space (not rushed)
- [ ] **Flow Coherence**: Reading all 3 layers feels like a complete journey

---

## Common Mistakes to Avoid

❌ **Treating layers as tone alternatives** ("Choose academic OR conversational")
- Reality: It's a sequential flow, all 3 layers required

❌ **Rushing the poetry** (1 paragraph Layer 1)
- Reality: Need 2-4 full paragraphs to establish emotional connection

❌ **Oversimplifying Layer 2** ("Make it accessible")
- Reality: Layer 2 is for technical depth; Layer 3 handles accessibility

❌ **Layer 3 just repeating Layer 2** in simpler words
- Reality: Layer 3 must add practical value (examples, code, real scenarios)

❌ **Using 3-layer for API docs**
- Reality: Internal/dev docs need precision, not poetry

❌ **Compressing all 3 layers into minimal space**
- Reality: Each layer needs room to breathe and accomplish its purpose

## TL;DR for Absolute Beginners

**Public-facing content introducing concepts?**
→ Use 3-Layer System: Poetic (2-4 para) → Academic (6+ para) → User-Friendly (examples)

**Internal dev docs or API reference?**
→ Skip poetry, use pure technical precision

**High verbosity = good**
→ Better deep treatment of fewer concepts than rushed treatment of many

**Remember**: It's a narrative JOURNEY (Layer 1 → 2 → 3), not a tone MENU
